# Anvilry v2.3 → v2.5 Upgrade Design

**Date:** 2026-06-23
**Phases:** v2.3.0 AI Transparency · v2.4.0 Performance · v2.5.0 Discoverability
**Branch model:** Each phase is a separate feature branch → PR to `develop`

---

## Context

Anvilry is a Next.js 16 App Router personal portfolio with four client-side views (Classic SSG, Gamified 3D graph, Chat AI, Developer terminal). This spec covers a three-phase upgrade derived from a deep-research audit (111 agents, 28 sources, 14 confirmed findings).

**Key constraints discovered during design:**
- The chatbot uses the raw `@anthropic-ai/sdk` + `@anthropic-ai/bedrock-sdk`, NOT the Vercel AI SDK — all AI features must work against the Anthropic SDK streaming surface.
- PPR (`cacheComponents: true`) is blocked by segment configs in 14 files. However, only 5 of these are page/special routes that need PPR migration — the remaining 9 are API routes (`runtime = "nodejs"`) which are request handlers and are unaffected by PPR.
- The existing `llms.txt`, `json-ld.tsx` (PersonJsonLd, BreadcrumbList, SoftwareSourceCode, FAQPage, ProfilePage, WebSiteJsonLd), and `buildLlmsTxt()` are already well-built. Phase 3 is additive only.

## Phase 1 — v2.3.0: AI Transparency

### Goal

Make Claude's reasoning visible in the chat UI. Hiring managers see a live "Thinking…" indicator while Claude reasons, then a collapsed "▶ Show reasoning" toggle once the answer arrives — real extended thinking, not a fake spinner.

### Feature Flag

| Variable | Location | Default | Effect |
|---|---|---|---|
| `EXTENDED_THINKING` | Server env var | `"true"` (enabled) | Set to `"false"` to disable extended thinking entirely — falls back to standard generation |
| `NEXT_PUBLIC_EXTENDED_THINKING` | Client env var | `"true"` | Mirrors the server flag so `<ThinkingBlock>` knows whether to render the toggle at all |

Both vars must be set together in Vercel dashboard. The `EXTENDED_THINKING !== "false"` check (not `=== "true"`) means the feature is opt-out, not opt-in — correct for a default-enabled flag.

### Architecture — 5 files modified, 0 new files

#### `src/lib/llm-trace.ts`

Add a second sentinel constant alongside `TRACE_DELIMITER`:

```ts
export const TRACE_DELIMITER = "";       // existing — unchanged
export const THINKING_SENTINEL = ""; // new — signals "thinking started"
```

`THINKING_SENTINEL` is two non-printable bytes. The client checks for it as a prefix at the START of the stream only (before any text arrives). It is never emitted mid-stream.

Extend the trace frame type (used by `splitTrace` in `use-chat.ts`):

```ts
export type TraceFrame = {
  model: string;
  fellBack: boolean;
  traceId?: string;
  usage?: LlmUsage;
  ttftMs?: number;
  latencyMs?: number;
  reasoning?: string;   // new — buffered thinking text, present when extended thinking ran
};
```

#### `src/lib/llm.ts`

Add `extendedThinking?: boolean` to `streamWithFallback`'s options parameter.

When `extendedThinking: true`:
1. Add `betas: ["interleaved-thinking-2025-05-14"]` to the Anthropic API call headers.
2. Add `thinking: { type: "enabled", budget_tokens: 1024 }` to the message params.
3. Emit `THINKING_SENTINEL` as the first bytes of the stream (before any content) — this tells the client to show the "Thinking…" animation immediately.
4. Handle `thinking_delta` events in the `for await` loop: buffer them into a `reasoningBuffer: string`. These arrive BEFORE `text_delta` events. Do NOT emit them as stream bytes — they are buffered server-side.
5. At clean finish: include `reasoning: reasoningBuffer` in the existing trace frame JSON (after `TRACE_DELIMITER`). The existing `traceFrame()` helper gets a `reasoning?: string` param.

**Model constraint:** Extended thinking requires `claude-sonnet-4-6` or higher (Haiku 4.5 does not support it). The fallback chain stays `Sonnet → Opus → Haiku`, but `extendedThinking` is disabled automatically when falling back to Haiku. Add a guard: if `model.includes("haiku")`, skip the thinking params silently.

**Token budget:** `budget_tokens: 1024` (conservative — keeps cost low on a public portfolio). The model may use fewer tokens; it never exceeds this budget.

#### `src/app/api/chat/route.ts`

Add one line after the rate-limit check:

```ts
const extendedThinking = process.env.EXTENDED_THINKING !== "false";
```

Pass it to `streamWithFallback`:

```ts
const stream = streamWithFallback(
  { ...params },
  { ..., extendedThinking },
);
```

#### `src/components/chat/use-chat.ts`

**`ChatMessage` type** — add two fields:

```ts
export type ChatMessage = {
  role: ChatRole;
  content: string;
  model?: string;
  fellBack?: boolean;
  reasoning?: string;    // buffered thinking text from trace frame
  isThinking?: boolean;  // true while THINKING_SENTINEL received but answer not yet started
};
```

**`splitTrace`** — already parses the trace frame JSON. Reads the new `reasoning` field naturally (no change needed to the function body — `JSON.parse(rest)` spreads all fields).

**Stream reading loop** — add THINKING_SENTINEL detection at the top of the `for await` read loop. When the first chunk arrives and starts with `THINKING_SENTINEL`:
1. Strip the sentinel bytes from the accumulated string.
2. Set `isThinking: true` on the in-progress assistant message.

When text content starts arriving (`acc` has non-empty text after sentinel strip):
1. Set `isThinking: false`.

#### `src/components/chat/chat-messages.tsx`

**`ThinkingBlock` component** (add inline, ~30 lines):

- **While `isThinking && isStreaming`**: render an animated "Thinking…" indicator. Use the existing `SkeletonMarkdownLine` pattern — three pulsing dots (`animate-pulse`) inside the same rounded-2xl bubble style as the existing "Thinking…" placeholder (line 197-200 of the current file).
- **After streaming completes and `reasoning` is present**: replace the animation with a static toggle button: `▶ Show reasoning (N words)` where N is `reasoning.split(' ').length`. Clicking expands an inline `<details>`/`<summary>` block.
- **Expanded state**: a `max-h-48 overflow-y-auto` scrollable `<pre>` block with `font-mono text-xs text-fg-subtle border-l-2 border-accent/30 pl-3` — matches the terminal aesthetic, visually distinct from the answer.
- **Only rendered when `NEXT_PUBLIC_EXTENDED_THINKING !== "false"`** — the env var check prevents the toggle from appearing on deployments where thinking is disabled.

**Integration point**: The existing empty-content branch (`segments.length === 0 && isStreaming && isLast`, lines 196-200) currently shows a static "Thinking…" text. Replace it with `<ThinkingBlock isThinking={m.isThinking} reasoning={m.reasoning} isStreaming={isStreaming && isLast} />`. This component handles both the "waiting" and "expandable" states.

### What is explicitly deferred to v2.3.1

- **Multi-modal file attachments** — requires content-length limit increase (64KB → ~2MB), file picker UI, CSP `img-src` updates, and server-side file handling. Separate PR.

### Testing

- `llm.test.ts`: add fixture for `THINKING_SENTINEL` prefix + `reasoning` field in trace frame JSON.
- `llm-trace.test.ts`: verify `THINKING_SENTINEL` is two bytes ``, different from `TRACE_DELIMITER`.
- Manual: set `EXTENDED_THINKING=true` in `.env.local`, ask a technical question ("how does the 3D graph work?"), verify reasoning trace appears and collapses correctly.

## Phase 2 — v2.4.0: Performance (PPR + use cache)

### Goal

Enable Partial Prerendering (`cacheComponents: true`) — static shells prerendered at build, dynamic content streamed at request time. Reduces TTFB for all four views. Also enables streaming metadata (automatic in Next.js 16 once PPR is on).

### Blocker Analysis

The `next.config.ts` comment references "17 files" — actual count is **14 files** with segment configs:

| Category | Files | Action |
|---|---|---|
| Page/special routes with `dynamic` | 5 files | Migrate to `use cache` — these block PPR |
| API routes with `runtime = "nodejs"` | 9 files | **No change needed** — API routes are request handlers, not pages; PPR does not apply |

**Files that need migration (5):**

| File | Current config | Migration |
|---|---|---|
| `src/app/llms.txt/route.ts` | `dynamic = "force-static"` | Remove; the route handler is already static (reads build-time data) — Next.js 16 treats GET handlers with no dynamic dependencies as static by default |
| `src/app/feed.xml/route.ts` | `dynamic = "force-static"` | Remove — same as above |
| `src/app/mcp/page.tsx` | `dynamic = "force-static"` | Remove; add `use cache` to any data-fetching functions inside the component |
| `src/app/api/resume.json/route.ts` | `dynamic = "force-static"` | Remove — static JSON handler, no dynamic dependencies |
| `src/app/admin/telemetry/page.tsx` | `runtime = "nodejs"` + `dynamic = "force-dynamic"` | Keep `runtime = "nodejs"` (legitimately needs Node.js for `timingSafeEqual`); remove `dynamic = "force-dynamic"` and replace with `noStore()` call inside the data-fetching function (opt out of caching at the data level, not the route level) |

**API routes that stay unchanged (9):**
`api/chat`, `api/mcp/[transport]`, `api/visit`, `api/github/stats`, `api/tts`, `api/error`, `api/tts-google`, `api/transcribe`, `api/cron/eval` — all have `runtime = "nodejs"` for legitimate reasons (streaming, AWS SDK, Node.js crypto). These are not pages and are not affected by `cacheComponents`.

### Architecture — 6 files modified

#### `next.config.ts`

Enable PPR by uncommenting the blocked config:

```ts
experimental: {
  inlineCss: true,
  viewTransition: true,
  cacheComponents: true,   // ← uncomment this
},
```

Remove the comment block explaining the blocker (it no longer applies after Phase 2 migration).

#### `src/app/layout.tsx`

Wrap the static shell (nav, footer, JSON-LD scripts) with `"use cache"` at the top of the file. The four view components (`ChatView`, `GamifiedView`, `DeveloperView`, `ClassicView`) are client-rendered behind Suspense boundaries — they remain dynamic (the "donut pattern").

```ts
"use cache";  // at top of layout.tsx
```

Only add this if the layout has no dynamic dependencies (auth, cookies, headers). Verify before applying.

#### The 5 page/special route files

Remove `export const dynamic = "force-static"` from the 4 static routes. For `admin/telemetry/page.tsx`: remove `dynamic = "force-dynamic"`, keep `runtime = "nodejs"`, add `import { unstable_noStore as noStore } from "next/cache"; noStore();` inside the data-fetching section.

### Risk Management

- Each file migration is a separate atomic commit.
- `pnpm build` is the gate after each commit — PPR errors surface as build-time errors, not runtime failures.
- If any route breaks, revert that single file and move on.
- The `admin/telemetry` page is the highest-risk change — test it locally before committing.

### Testing

- `pnpm build` must complete without PPR errors after all 5 migrations.
- Verify `/llms.txt`, `/feed.xml`, and `/api/resume.json` still return correct content.
- Verify `/admin/telemetry` still loads and shows live Redis data (dynamic, not cached).
- Verify the 9 unchanged API routes still work (spot-check `/api/chat`, `/api/tts`).

## Phase 3 — v2.5.0: Discoverability

### Goal

Two additive upgrades: (1) per-page `.md` variants so AI assistants can fetch clean markdown for any content page, (2) minor JSON-LD enhancement for structured skills data.

### Architecture — 5 files modified, 4 new files

#### New: Per-page `.md` route handlers (4 new files)

For each content collection, add a sibling route at `[slug].md/route.ts`:

```
src/app/articles/[slug].md/route.ts
src/app/notes/[slug].md/route.ts
src/app/work/[slug].md/route.ts
src/app/projects/[slug].md/route.ts
```

Each handler:
1. Looks up the item by slug from the relevant `all*` collection in `src/lib/content.ts`.
2. Returns 404 if not found.
3. Returns `text/markdown; charset=utf-8` with the MDX body — frontmatter stripped, raw markdown body only.
4. Marked `force-static` (or simply no dynamic deps — pure content lookup).

**Why MDX body only, no frontmatter:** The llms.txt spec asks for "clean markdown" — frontmatter YAML clutters the signal for AI assistants. The structured metadata is already in the JSON-LD on the HTML page.

#### `src/lib/llms-txt.ts` — extend `buildLlmsTxt()`

After the existing sections, add a new `## Markdown Versions` H2 section with links to each `.md` route:

```
## Markdown Versions
Individual pages are available as clean markdown by appending .md to any URL:
- Work: https://anvilry.vercel.app/work/[slug].md
- Projects: https://anvilry.vercel.app/projects/[slug].md
- Notes: https://anvilry.vercel.app/notes/[slug].md
- Articles: https://anvilry.vercel.app/articles/[slug].md
```

This tells AI crawlers exactly how to fetch any page in markdown. The section is generated from the actual content collections (same as the existing sections) so it never drifts.

#### `src/components/json-ld.tsx` — `PersonJsonLd` update

The existing `PersonJsonLd` already has `knowsAbout: skills.flatMap((s) => s.items)` (plain string array). Add a `skills` property alongside it using `DefinedTerm[]` for the machine-readable structured form (the research-verified 2-1 finding):

```ts
skills: skills.flatMap((s) => s.items).map((item) => ({
  "@type": "DefinedTerm",
  name: item,
})),
```

**Why both `knowsAbout` and `skills`:** `knowsAbout` (Text[]) is broadly supported; `skills` (DefinedTerm[]) is schema.org ext-pending but interpreted by AI crawlers. Additive — zero risk to the existing signal.

### Testing

- Fetch `https://localhost:3000/work/pensieve.md` — verify clean markdown body, no YAML frontmatter, correct `Content-Type`.
- Fetch `https://localhost:3000/notes/nonexistent.md` — verify 404.
- Inspect `PersonJsonLd` output in page source — verify both `knowsAbout` (strings) and `skills` (DefinedTerm objects) are present.
- Verify `/llms.txt` includes the new `## Markdown Versions` section.
